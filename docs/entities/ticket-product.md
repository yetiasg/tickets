# Ticket Product

## Purpose

`ticket_products` stores the tenant-defined ticket product that can later be sold and transformed into customer-facing ticket instances.

This table represents the business object created by a tenant user before a customer places an order.

## Core Fields

1. `id`: product identifier generated by the application.
2. `tenant_id`: tenant ownership boundary.
3. `event_id`: event to which the product belongs.
4. `ticket_template_id`: optional template used for generation.
5. `name`: business-visible product name.
6. `code`: tenant-scoped business code.
7. `status`: product lifecycle state.
8. `delivery_type`: delivery form supported by the product.
9. `available_quantity`: number of units available for sale.
10. `additional_info`: explicit product-level business metadata.
11. `max_validation_count`: product-level validation limit.
12. `is_single_use`: whether a successful validation consumes the ticket instance.
13. `valid_from` and `valid_until`: optional default validation window.
14. `validation_rules`: extensible validation configuration payload.

## Invariants

1. `available_quantity` must not be negative.
2. `max_validation_count` must be at least `1`.
3. `valid_until` must not be earlier than `valid_from`.
4. `code` must be unique inside the same tenant.

## Notes

1. This is the first persistence slice only.
2. Inventory, customer ticket instances, validation attempts, and lifecycle events should be modeled separately in later iterations.
3. The current table intentionally keeps enough information to represent tenant-created ticket products before the rest of the ordering flow is implemented.
4. The module keeps a dedicated domain object in `src/ticket-product/core/domain/ticket-product.ts`.
5. The module keeps a dedicated persistence entity in `src/ticket-product/core/adapter/ticket-product.entity.ts`.
